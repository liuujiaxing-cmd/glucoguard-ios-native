# 技术架构与数据隐私说明文档

## 1. 项目架构概述 (Architecture Overview)

SugarGuard (以及未来的公益 App 系列) 采用 **"本地优先 (Local-First)"** 的技术架构。这意味着所有的核心功能和数据存储都优先在用户设备上完成，仅在必要时（如 AI 分析、云同步）才与服务器交互。

### 1.1 前端 (Frontend / Client)
*   **技术栈**: 原生 iOS (SwiftUI)
*   **优势**:
    *   **高性能**: 启动速度快，滑动流畅，无白屏。
    *   **原生体验**: 完全符合 Apple 的设计规范 (Human Interface Guidelines)。
    *   **隐私安全**: 代码编译为二进制文件，比网页更难被篡改。

### 1.2 后端 (Backend)
*   **当前状态**: **无服务器 (Serverless) / 本地模式**。
*   **说明**: 目前 V1.0 版本没有传统的后端数据库。所有的业务逻辑（如计算平均血糖、生成建议）都在用户手机的芯片上直接运行。
*   **未来扩展**: 后期如果需要云同步功能，会接入 Apple 的 **CloudKit** (利用用户的 iCloud 空间存储) 或轻量级的云数据库 (如 Supabase)。

## 2. 数据存储方案 (Data Storage)

用户的隐私是我们最看重的，因此我们采用了以下存储策略：

### 2.1 核心健康数据 (Core Health Data)
*   **存储位置**: 用户手机本地沙盒 (Local Sandbox)。
*   **技术实现**: `UserDefaults` (轻量级) 或 `CoreData / SwiftData` (结构化数据)。
*   **安全性**: 只有 App 本身能读取这些文件，其他 App 无法访问。
*   **内容**: 血糖记录、用药记录、个人设置。

### 2.2 敏感信息 (Sensitive Info)
*   **存储位置**: 暂未收集。
*   **说明**: 我们**不收集**用户的手机号、身份证、真实姓名等实名信息。App 内显示的“用户 8802”仅为本地生成的随机 ID。

## 3. 关于官网制作 (Website Guide)

既然你计划制作一系列公益 App，建立一个统一的官网品牌非常有必要。

### 3.1 官网内容规划
建议官网包含以下板块：
1.  **首页 (Home)**: "科技向善，守护健康" —— 品牌愿景介绍。
2.  **App 矩阵 (Products)**:
    *   SugarGuard (血糖管家)
    *   HeartGuard (心脏守护 - 规划中)
    *   MindCare (心理抚慰 - 规划中)
3.  **公益透明 (Transparency)**: 公示项目运营成本、广告收入去向（如捐赠给糖尿病基金会或支付服务器费）。
4.  **加入我们/捐赠 (Support Us)**: 招募开发者志愿者，或接受小额捐赠。

### 3.2 制作工具推荐
*   **方案 A (最简单 - 推荐)**: 使用 **Notion + Super.so**。你只需要写 Notion 笔记，自动生成漂亮的网站。
*   **方案 B (最专业)**: 使用 **WordPress** 或 **Framer**。拖拽式建站，设计感强。
*   **方案 C (程序员风格)**: 使用 **GitHub Pages + Jekyll/Hugo**。完全免费，托管在 GitHub 上。

### 3.3 域名建议
建议注册一个 `.org` (非营利组织) 后缀的域名，例如：
*   `healthguard.org`
*   `openhealth.org`
*   `公益科技.cn`
